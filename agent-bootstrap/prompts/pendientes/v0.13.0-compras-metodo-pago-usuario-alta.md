# v0.13.0 — Métodos de pago en Compras

**VERSIÓN:** v0.13.0  
**SLUG:** compras-metodo-pago-usuario-alta  
**DEPENDENCIAS:** modelo de `Compra` y módulo de seguridad/usuarios ya implementados  
**ESTIMACIÓN:** 2h  
**PRIORIDAD:** alta

---

## Origen

> Queremos mejorar el registro de las compras:
> - Eliminar el campo `numeroFactura` (ya no se va a usar).
> - Registrar el **método de pago** y, cuando sea tarjeta, la compañía de la tarjeta y los últimos 4 dígitos.
> - Registrar **qué usuario** fue el que dio de alta la compra, usando la autenticación existente.

---

## Descripción

Modificar la entidad `Compra` y su API para:

- Eliminar el campo `numeroFactura`.
- Agregar campos para el método de pago y, si corresponde, datos de tarjeta.
- Guardar automáticamente el usuario autenticado que creó la compra.
- Exponer los nuevos datos en los endpoints de creación, actualización y listado/detalle de compras.

## PASO 1 — Actualizar modelo y BD de `Compra`

1. En la entidad `Compra`:
   - **Eliminar** el campo `numeroFactura` (propiedad, getter/setter, anotaciones JPA).
   - **Agregar**:
     - `metodoPago`:
       - Crear un enum `MetodoPago` (por ejemplo: `EFECTIVO`, `TRANSFERENCIA`, `TARJETA`, `OTRO`) o reutilizar uno existente si ya hay algo similar.
       - Mapearlo en JPA con `@Enumerated(EnumType.STRING)`.
     - `companiaTarjeta` (`String`, longitud razonable, ej. 50).
     - `ultimos4Tarjeta` (`String` o `char(4)`, almacenar exactamente 4 caracteres).
     - `usuarioAlta`:
       - Relación `@ManyToOne` con `Usuario`.
       - Campo no nulo para nuevas compras (permitir `null` solo para datos históricos si hace falta).
2. Ajustar el mapeo JPA (columnas, `nullable`, longitudes) y cualquier `equals/hashCode` o `toString` que haga referencia a estos campos.
3. Si se usan scripts SQL o migraciones manuales:
   - **ALTER TABLE compras**:
     - Dropear columna `numero_factura`.
     - Agregar columnas para `metodo_pago`, `compania_tarjeta`, `ultimos4_tarjeta`, `usuario_alta_id` (con `FOREIGN KEY` a `usuarios`).
   - Documentar cambios brevemente en la sección correspondiente de `ESTRUCTURA_BD_ENTIDADES.md`.

**Verificación:** la app arranca, Hibernate no reporta errores de mapeo, y la tabla de compras refleja las nuevas columnas sin `numero_factura`.

---

## PASO 2 — DTOs y validaciones de método de pago

1. En los DTOs / request objects para creación y actualización de compra:
   - Eliminar cualquier referencia a `numeroFactura`.
   - Agregar campos:
     - `metodoPago` (string o enum compatible con `MetodoPago`).
     - `companiaTarjeta` (opcional).
     - `ultimos4Tarjeta` (opcional).
2. En el servicio de compras (o en el controlador, según dónde esté la lógica):
   - Al crear o actualizar:
     - Mapear `metodoPago` del DTO al enum `MetodoPago`.
     - Aplicar reglas:
       - Si `metodoPago == TARJETA`:
         - `companiaTarjeta` **obligatorio** (no vacío).
         - `ultimos4Tarjeta` **obligatorio**, exactamente 4 dígitos (`^[0-9]{4}$`).
         - Si la validación falla → lanzar excepción de negocio y devolver 400 Bad Request con mensaje descriptivo.
       - Si `metodoPago != TARJETA`:
         - `companiaTarjeta` y `ultimos4Tarjeta` se deben guardar como `null` (aunque el cliente los envíe, ignorarlos).
3. Actualizar cualquier mapeo entre entidad y DTO de respuesta para incluir los nuevos campos.

**Verificación:** crear compras vía API con:
- `metodoPago = EFECTIVO` (sin datos de tarjeta) → OK.
- `metodoPago = TARJETA` sin `companiaTarjeta` o sin `ultimos4Tarjeta` → 400 con mensaje claro.
- `metodoPago = TARJETA` con `ultimos4Tarjeta` distinto de 4 dígitos → 400.

## PASO 3 — Registrar el usuario que creó la compra

1. En el endpoint de **creación de compra** (`POST /compras`):
   - Obtener el principal autenticado desde `Authentication` (por ejemplo `AuthUser`).
   - Con ese `id` o `email`, cargar la entidad `Usuario` correspondiente.
   - Setear `compra.setUsuarioAlta(usuario)` antes de guardar.
   - No permitir que el cliente envíe el `usuarioAlta` en el cuerpo: el valor siempre debe venir del contexto de seguridad.
2. En el endpoint de **actualización**:
   - Mantener el `usuarioAlta` original (no reescribirlo con el usuario que edita).
3. En las respuestas de compra (detalle y listado):
   - Incluir datos de `usuarioAlta` de forma amigable (por ejemplo):
     - `usuarioAltaId`
     - `usuarioAltaEmail`
     - opcionalmente `usuarioAltaNombre`.

**Verificación:** al crear una compra autenticado como `admin@admin.com`, la fila en BD tiene `usuario_alta_id` apuntando al usuario admin, y la respuesta devuelve sus datos.
