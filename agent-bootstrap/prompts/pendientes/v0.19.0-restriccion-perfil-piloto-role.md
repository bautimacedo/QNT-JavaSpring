# v0.19.0 — Restringir "Perfil Piloto" exclusivamente a ROLE_PILOTO

**VERSIÓN:** v0.19.0
**SLUG:** restriccion-perfil-piloto-role
**DEPENDENCIAS:** ninguna (MiPerfilPilotoController ya existe)
**ESTIMACIÓN:** 1h
**PRIORIDAD:** alta

---

## Descripción

Actualmente, `PUT /mi-perfil` tiene `@PreAuthorize("isAuthenticated()")`, lo que permite que **cualquier usuario autenticado** (incluido ROLE_USER y ROLE_ADMIN sin rol piloto) pueda enviar y guardar campos específicos de piloto como `passwordMission`. Además, el DTO `ActualizarMiPerfilRequest` no incluye `horasVuelo` ni `cantidadVuelos`, que son datos que un piloto debería poder actualizar desde su perfil.

Esta tarea tiene dos objetivos:
1. **Agregar `horasVuelo` y `cantidadVuelos`** al DTO de actualización de perfil para que el piloto pueda modificarlos desde la vista "Perfil Piloto".
2. **Restringir los campos de piloto**: si el usuario no tiene ROLE_PILOTO ni ROLE_ADMIN, los campos `passwordMission`, `horasVuelo` y `cantidadVuelos` deben ser ignorados (o devolver 403 si solo se enviaron esos campos). Los campos generales (nombre, apellido, dni) siguen accesibles para cualquier usuario autenticado.

## PASO 2 — Restringir campos de piloto en PUT /mi-perfil

En `MiPerfilPilotoController.actualizarMiPerfil()`, agregar la verificación de rol antes de aplicar campos de piloto:

```java
@PutMapping
@Transactional
@PreAuthorize("isAuthenticated()")
public ResponseEntity<?> actualizarMiPerfil(Authentication authentication,
                                             @RequestBody ActualizarMiPerfilRequest request) {
    AuthUser auth = authUser(authentication);
    try {
        Usuario usuario = usuarioBusiness.load(auth.getId());

        // Campos generales — cualquier usuario autenticado puede modificarlos
        if (request.nombre() != null && !request.nombre().isBlank()) usuario.setNombre(request.nombre());
        if (request.apellido() != null) usuario.setApellido(request.apellido().isBlank() ? null : request.apellido());
        if (request.dni() != null) usuario.setDni(request.dni().isBlank() ? null : request.dni());

        // Campos exclusivos de piloto — solo ROLE_PILOTO o ROLE_ADMIN pueden modificarlos
        if (isPilotoOrAdmin(authentication)) {
            if (request.passwordMission() != null) {
                String pm = request.passwordMission().trim();
                if (pm.length() > 30) pm = pm.substring(0, 30);
                usuario.setPasswordMission(pm.isBlank() ? null : pm);
            }
            if (request.horasVuelo() != null) usuario.setHorasVuelo(request.horasVuelo());
            if (request.cantidadVuelos() != null) usuario.setCantidadVuelos(request.cantidadVuelos());
        }
        // Si el usuario no es piloto/admin y envió campos de piloto, simplemente se ignoran (sin error).

        usuarioBusiness.update(usuario);
        return ResponseEntity.ok(usuario);
    } catch (NotFoundException e) {
        return ResponseEntity.notFound().build();
    } catch (FoundException e) {
        return ResponseEntity.status(HttpStatus.CONFLICT).body(e.getMessage());
    } catch (BusinessException e) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).build();
    }
}
```

El método `isPilotoOrAdmin(authentication)` ya existe en el controller.

**Verificación:**
- Usuario con ROLE_USER que envía `passwordMission` → el campo es ignorado, el resto se guarda.
- Usuario con ROLE_PILOTO que envía `horasVuelo=100` → se guarda correctamente.
- Usuario con ROLE_PILOTO que envía `nombre` y `horasVuelo` → ambos se guardan.

## Notas

- Se decidió **ignorar silenciosamente** los campos de piloto para usuarios no-piloto en lugar de devolver 403, para no romper clientes que puedan estar enviando esos campos sin querer. Si en el futuro se prefiere un error explícito, cambiar el comportamiento.
