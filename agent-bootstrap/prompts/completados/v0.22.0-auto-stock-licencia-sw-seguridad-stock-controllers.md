**DEPENDENCIAS:** v0.18.0 (auto-inventario al comprar equipo)

**PRIORIDAD:** alta

## Contexto y estado actual

Al registrar una compra de tipo EQUIPO (DRON, BATERIA, HELICE), el sistema ya crea
automáticamente el ítem en el inventario con estado NO_LLEGO. Esta lógica vive en
`CompraBusiness.crearItemEnInventario()` y fue implementada en v0.18.0.

Lo que **falta** es:
1. Compra de tipo `LICENCIA_SW` → **no crea ninguna `Licencia`** automáticamente.
   El usuario debe ir a stock/licencias y crearla a mano, sin vinculación a la compra.
2. Los controllers de stock (`DronRestController`, `BateriaRestController`,
   `HeliceRestController`, y otros) **no tienen `@PreAuthorize`** — cualquier cliente
   sin autenticar puede invocar GET/POST/PUT/DELETE sobre el inventario.

## PASO 3 — Implementar crearLicenciaSWDesdeCompra()

Agregar el método privado en `CompraBusiness.java`:

```java
/**
 * Crea automáticamente una Licencia (SW) en stock al registrar una compra de tipo LICENCIA_SW.
 * La licencia queda vinculada a la compra mediante compra_id para trazabilidad completa.
 * El nombre inicial se toma de descripcion; si está vacío se usa "Licencia SW".
 */
private void crearLicenciaSWDesdeCompra(Compra compra, CreateCompraRequest request)
        throws BusinessException {
    Licencia licencia = new Licencia();

    String nombre = (request.descripcion() != null && !request.descripcion().isBlank())
            ? request.descripcion().trim()
            : "Licencia SW";
    licencia.setNombre(nombre);
    licencia.setCompra(compra);
    licencia.setFechaCompra(compra.getFechaCompra());
    licencia.setActivo(true);
    // numLicencia, caducidad y version quedan null: el admin los completa desde stock/licencias

    licenciaBusiness.add(licencia);
    log.info("Licencia SW '{}' creada automáticamente a partir de compra id={}", nombre, compra.getId());
}
```

**Notas:**
- `licencia.setCompra(compra)` vincula la licencia a la compra. El campo ya existe en la
  entidad (`@ManyToOne Compra compra`) — no requiere migración.
- `numLicencia`, `caducidad` y `version` quedan en null: el admin los completa desde el
  listado de licencias cuando llegue la documentación.
- No llamar a este método en `update()`: si se edita una compra existente, no se dupliquen licencias.

## PASO 4 — Seguridad: agregar @PreAuthorize a los controllers de stock

Los siguientes controllers **no tienen `@PreAuthorize`** en ningún método.
Agregar las anotaciones correspondientes en cada uno.

### Regla general de roles para stock:

| Operación | Roles requeridos |
|-----------|-----------------|
| GET (listar, obtener por id) | `isAuthenticated()` |
| POST (crear) | `hasRole('ADMIN') or hasRole('USER')` |
| PUT (actualizar) | `hasRole('ADMIN') or hasRole('USER')` |
| DELETE (eliminar) | `hasRole('ADMIN')` |

Aplicar esta regla a:
- `DronRestController` — métodos: `list()`, `load()`, `add()`, `update()`, `delete()`
- `BateriaRestController` — mismos métodos
- `HeliceRestController` — mismos métodos
- `DockRestController` — mismos métodos (verificar si tiene delete)
- `AntenaRtkRestController` — mismos métodos
- `AntenaStarlinkRestController` — mismos métodos
- `AccesorioRestController` — mismos métodos

**Importante:** También agregar `@PreAuthorize("isAuthenticated()")` a nivel de clase
(`@RestController` + `@PreAuthorize`) donde todos los métodos sean al menos autenticados,
y sobreescribir en los métodos específicos que requieran un rol más restrictivo (POST/PUT/DELETE).
Usar la anotación a nivel de método para claridad.
