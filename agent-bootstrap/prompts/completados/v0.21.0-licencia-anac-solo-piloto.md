# v0.21.0 — Restricción ROLE_PILOTO obligatorio para licencias ANAC

**VERSIÓN:** v0.21.0
**SLUG:** licencia-anac-solo-piloto
**DEPENDENCIAS:** v0.20.0
**ESTIMACIÓN:** 45min
**PRIORIDAD:** alta

---

## Objetivo

Hoy los endpoints de licencias ANAC aceptan `hasRole('PILOTO') or hasRole('ADMIN')`.
Eso permite que un usuario con solo ROLE_ADMIN cree/edite/elimine licencias de piloto en
su propio perfil, lo cual es incorrecto: **un usuario debe tener ROLE_PILOTO obligatoriamente**
para gestionar licencias ANAC, con o sin ROLE_ADMIN.

La corrección es en dos capas:
1. **Controller** — cambiar el `@PreAuthorize` de los 8 endpoints de licencias.
2. **Business** — agregar validación de rol en `LicenciaANACBusiness.add()` para que
   ninguna ruta de código (presente o futura) pueda crear una licencia ANAC para un
   usuario sin ROLE_PILOTO.

## PASO 1 — Controller: cambiar @PreAuthorize en los 8 endpoints de licencias

En `MiPerfilPilotoController.java`, los siguientes métodos tienen:

```java
@PreAuthorize("hasRole('PILOTO') or hasRole('ADMIN')")
```

Cambiar **todos** a:

```java
@PreAuthorize("hasRole('PILOTO')")
```

Los métodos afectados son exactamente estos (buscar por la anotación):

| Método HTTP | Ruta |
|-------------|------|
| GET | `/mi-perfil/licencias` |
| POST | `/mi-perfil/licencias` |
| PUT | `/mi-perfil/licencias/{id}` |
| DELETE | `/mi-perfil/licencias/{id}` |
| PUT | `/mi-perfil/licencias/{id}/imagen-cma` |
| GET | `/mi-perfil/licencias/{id}/imagen-cma` |
| PUT | `/mi-perfil/licencias/{id}/imagen-certificado-idoneidad` |
| GET | `/mi-perfil/licencias/{id}/imagen-certificado-idoneidad` |

**NO tocar** los endpoints de perfil general (`GET /mi-perfil`, `PUT /mi-perfil`,
`PUT/GET /mi-perfil/foto-perfil`, `PUT /mi-perfil/cambio-password`).

## PASO 2 — Controller: corregir el helper isPilotoOrAdmin para mostrar licencias

En `getMiPerfil()` hay este bloque que muestra las licencias:

```java
if (isPilotoOrAdmin(authentication)) {
    // carga y pone licencias en el body
} else {
    body.put("licencias", List.of());
}
```

El helper `isPilotoOrAdmin` incluye ROLE_ADMIN, por lo que un admin sin ROLE_PILOTO
ve el campo `licencias` en su perfil (aunque sea una lista vacía, el contrato es incorrecto).

**Cambio:** reemplazar la condición por un check exclusivo de ROLE_PILOTO:

```java
boolean esPiloto = authentication.getAuthorities().stream()
        .anyMatch(a -> "ROLE_PILOTO".equals(a.getAuthority()));

if (esPiloto) {
    List<LicenciaANAC> licencias = licenciaANACBusiness.listByPiloto(auth.getId());
    body.put("licencias", licencias.stream()
            .map(this::licenciaToMap)
            .collect(Collectors.toList()));
} else {
    body.put("licencias", List.of());
}
```

El helper `isPilotoOrAdmin` sigue existiendo tal cual — se usa para los campos de perfil
de piloto (`passwordMission`, `horasVuelo`, `cantidadVuelos`) donde sí tiene sentido que
un admin pueda editarlos. No eliminarlo ni modificarlo.

## PASO 3 — Business: validar ROLE_PILOTO en LicenciaANACBusiness.add()

En `LicenciaANACBusiness.add(LicenciaANAC entity)`, agregar al inicio del bloque `try`
la siguiente validación:

```java
// Validar que el piloto asociado tiene ROLE_PILOTO
if (entity.getPiloto() == null) {
    throw new BusinessException("La licencia ANAC debe estar asociada a un usuario piloto");
}
boolean tienRolPiloto = entity.getPiloto().getRoles() != null &&
        entity.getPiloto().getRoles().stream()
                .anyMatch(r -> "ROLE_PILOTO".equals(r.getCodigo()));
if (!tienRolPiloto) {
    throw new BusinessException(
        "El usuario '" + entity.getPiloto().getEmail() +
        "' no tiene el rol PILOTO y no puede tener licencias ANAC");
}
```

**Notas importantes:**
- El `entity.getPiloto()` viene cargado desde el controller con `usuarioBusiness.load(auth.getId())`,
  que trae los roles eager. No hace falta un load adicional en el business.
- Si en algún otro punto del código se llama a `add()` con un Usuario cargado de forma lazy
  (roles vacíos), la excepción fallará igualmente protegiendo la integridad. Es el
  comportamiento correcto: quien llame a `add()` es responsable de pasar el entity con roles cargados.
- **No modificar** los métodos `update()`, `list()`, `load()` ni `delete()` de LicenciaANACBusiness.

## PASO 5 — Verificación manual

Probar con Postman/curl o un smoke test:

1. Usuario con solo ROLE_ADMIN:
   - `POST /mi-perfil/licencias` → debe responder **403 Forbidden**
   - `GET /mi-perfil/licencias` → debe responder **403 Forbidden**

2. Usuario con ROLE_PILOTO (con o sin ROLE_ADMIN):
   - `POST /mi-perfil/licencias` → debe responder **201 Created**
   - `GET /mi-perfil/licencias` → debe responder **200 OK**

3. Intentar llamar a `LicenciaANACBusiness.add()` con un Usuario sin ROLE_PILOTO
   (si hay un test existente o se puede verificar vía debug): debe lanzar `BusinessException`.
